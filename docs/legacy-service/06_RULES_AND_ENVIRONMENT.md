# Production Rules and Environment Definition

## 1. 기술 스택

현재 레거시:
- Frontend: Vanilla JavaScript
- Backend: FastAPI
- DB: Supabase 연동 구조
- Assets: 정적 HTML 게임 파일, badge 이미지, favicon, lang JSON

신규 목표:
- Frontend: React + TypeScript + Vite
- Backend: FastAPI 유지
- DB: 기존 스키마 유지 후 점진 확장

## 2. 개발 환경 원칙

- 정적 파일 캐시는 버전 문자열로 제어
- 언어 데이터는 `lang.json` 기준
- 게임 자산은 언어별 디렉토리 구조 유지
- 백엔드 API 계약은 1차 마이그레이션 동안 최대한 유지

## 3. 제작 규칙

### 3.1 기능 제작 규칙
- 레거시 기능 누락 없이 parity 우선
- 정책 변경이 아닌 경우 동작 단순화 금지
- 화면만 비슷하고 정책이 다른 구현 금지
- 카테고리명, 버튼명, 기능명, 안내문구, 팝업 문구, 정책 문구, 배지명 등 사용자 노출 텍스트는 레거시와 동일 유지
- 리팩터링 중 임의 wording 변경 금지
- 텍스트 변경은 별도 기획 변경으로만 처리
- 현재 단계 문서 범위를 벗어난 기능 확장 금지
- 현재 단계 완료 조건이 충족되면 해당 단계 구현을 종료하고 다음 단계로 이동
- 구현 중 떠오른 추가 아이디어는 즉시 반영하지 않고 backlog 또는 체크리스트 메모로만 남긴다
- parity 단계에서는 개선보다 동일성 우선

### 3.2 UI 제작 규칙
- 온보딩 UX 유지
- 마이페이지 별도 테마 유지
- 코멘트/댓글 인터랙션은 깜빡임 최소화
- 팝업은 기본 alert 금지

### 3.3 데이터 규칙
- 서버 검증 우선
- 닉네임/코멘트/댓글 escape 유지
- 배지 조건은 서버 계산 유지

### 3.4 문서화 규칙
- 레거시와 신규 차이는 명시적으로 문서화
- 신규 추가 기능은 별도 변경 목록으로 관리

## 4. 환경 변수

현행 기준:
- `SUPABASE_URL`
- `SUPABASE_KEY`
- `ADMIN_NICKNAMES`
- `ADMIN_PASSWORD`
- `CORS_ALLOW_ORIGINS`

## 5. 디렉토리 운영 원칙

레거시:
- `AI_Game_legacy`

신규:
- `AI_Game`

원칙:
- 두 시스템을 섞지 않는다.
- 신규는 별도 루트에서 구축한다.
- 마이그레이션 중 레거시는 기준선 참조용으로만 사용한다.

## 6. 품질 기준

- 주요 흐름 수동 테스트 가능해야 함
- 언어별 UI 회귀 없어야 함
- 댓글/배지/관리자 기능 회귀 없어야 함
- 정책 페이지 누락 없어야 함
